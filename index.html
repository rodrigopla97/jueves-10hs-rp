Areas de GIT

Working directory

Es la carpeta donde realizamos los cambios en nuestro proyecto, en este lugar rara vez utilizaremos la consola y es el área donde se visualizan todos los cambios de los archivos

  • mkdir <dir> crea un directorio con el nombre especificado. Por ejemplo para crear una carpeta deberiamos escribir mkdir proyecto-git
  • cd<dir> cambia el directorio actual por el indicado. Para acceder a una carpeta, la ruta debe ser correcta como cd proyecto-git
  • rm <dir>elimina el directorio o archivo indicado. Para eliminar un repositorio de git debemos utilizarlo de la siguiente manera rm - rf .git
  • git init inicia un repositorio de git o lo reinicia en caso de que ya exista. Debemos tener en cuenta que si utilizamos este comando en ubicaciones diferentes, cada ubicación se considerará un repositorio en sí

Staging Area (Index)

Es el área de control de cambios, donde se verfucan las modificaciones realizadas en los archivos del proyecto, Aquí se realizan los snapshots de codigo para luego ser confirmados.

  • git status verifica los cambios del WORKING DIRECTORY y los compara con lo almacenado en el INDEX. Cada archivo posee un estado segun las acciones realizadas.
    • untracked: no se está haciendo seguimiento.
    • tracked: se está haciendo seguimiento.
    • modified: difiere con lo que se encuentra en el index.
    • commited: los cambios estan registrados.
  • git add <file> agrega al INDEX los archivos especificados, también se pueden agregar directorios, siempre y cuando estos tengan archivos en su interior. Un uso rapido de este comando es git add . que agrega todos los archivos que se hayan modificado o creado en el directorio actual
  • git rm --cached <file> Quita el archivo del área de seguimiento pero lo mantiene en el WORKING DIRECTORY. Este comando se utiliza cuando no existe una referencia para reestablecer el contenido.
  • git restore <file> Quita el archivo del INDEX, y lo reestablece a como estaba en la última cabecera.

Repository (Local / Remote)

Es el área donde se almacena las capturas en formato BLOB (Binary Object File) a las cuales podemos acceder mediante el historial de cambios. Para quenerar estos archivos debemos emitir un commit, el mismo valida los cambios y los almacena en la base de datos de GIT.

  • git commit Abre el editor para generar una confirmación de cambios que será guardada en la base de datos del repositorio. Un uso rapido de este comando es git commit -m "message" que permite generar un commit con mensaje sin abrir el editor.
  • git diff <ref> Compara el WORKING DIRECTORY con el commit al cual apunta la referencia especificada. Una forma de comparar los últimos cambio realizados cuando no se hizo un commit es git diff HEAD, pero si queremos saber como difiere muestra cabecera con respecto a otro commit, podemos indicarlo de distintas maneras
    • refencia absoluta: especificamente hablando es el numero de HASH del commit que queremos comparar, por ejemplo 20d2160
    • referencia relativa: indica a partir de la cabecera actual, cuántos commit atrás está nuestra referencia, por ejemplo HEAD^^ indica dos ancestros antes de HEAD y HEAD^4 indica 4 commits antes del HEAD
  • git blame <file> muestra las líneas de código del archivo especificado, colocando al principio el número de commit, el nombre del autor del commit, la fecha y hora en que se realizó el mismo. Incluso indica cuando los cambios no fueron confirmados.
  • git remote lista los repositorios remotos que hayan sido agregados. Si nuestro proyecto es un clón por defecto se agrega un remoto llamado origin, preo si fue creado, debemos hacerlo manualmente usando el comando git remote add <alias>

Apuntadores de GIT

HEAD

Es un apuntador movil que nos permite navegar en el historial de cambios, normalmente apunta al último commit de la rama donde nos encontremos, y podemos cambiar su posición para poder visualizar el proyecto a como se encontraba en algún punto de referencia (commit) utilizando el comando git checkout <ref>

BRANCH

Las ramas son apuntadores dinámicos que siempre hacen referencia al último commit dentro del historial. Podemos crear diferentes ramas y cada una apuntara a un commit particular, este puede ser el mismo o pueden ser totalmente diferentes. Si se quiere comparar, podrían interpretarse como líneas de tiempo paralelas; las ramas son necesarias ya que permiten al apuntador HEAD navegar en el historial y volver al último commit sin problemas.
