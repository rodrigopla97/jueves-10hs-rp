Configuración Inicial

Para empezar a trabajar con Git debemos establecer una mínima configuración necesaria para realizar nuestros commits. Esto consiste en definir el nombre y correo de contacto, ademas de algunos parametros extra.

  • git config --global user.name "username" define el nombre de contacto con el que se asocian los commits
  • git config --global user.email "email" define el correo de contacto con el que se asocian los commits.
  • git config core.editor "path | editor" define el editor de código predeterminado para la operaciones con git.
  • git config alias.value "comand" define un atajo para algún comando de git.

Areas de GIT

Working directory

Es la carpeta donde realizamos los cambios en nuestro proyecto, en este lugar rara vez utilizaremos la consola y es el área donde se visualizan todos los cambios de los archivos.

  • mkdir <dir> crea un directorio con el nombre especificado. Por ejemplo para crear una carpeta deberiamos escribir mkdir proyecto-git
  • cd <dir> cambia el directorio actual por el indicado. Para acceder a una carpeta, la ruta debe ser correcta como cd proyecto-git
  • rm <dir> elimina el directorio o archivo indicado. Para eliminar un repositorio de git debemos utilizarlo de la siguiente manera rm -rf .git
  • git init inicia un repositorio de git o lo reinicia en caso que este ya exista. Debemos tener en cuenta que si utilizamos este comando el ubicaciones diferentes, cada ubicacion se considerara un repositorio en si.

Staging Area (Index)

Es el área de control de cambios, donde se verifican las modificaciones realizadas en los archivos del proyecto. Aqui se realizan los snapshots de código para luego ser confirmados.

  • git status verifica los cambios del WORKING DIRECTORY y los compara con lo almacenado en el INDEX. Cada archivo posee un estado segun las acciones realizadas
    • untracked: no se está haciendo seguimiento
    • tracked: se está haciendo seguimiento
    • modified: difiere con lo que se encuentra en el index
    • commited: los cambios están registrados
  • git add <file> agrega al INDEX los archivos especificados, tambien se pueden agregar directorios, siempre y cuando estos tengan archivos en su interior. Un uso rapido de este comando es git add . que agrega todos los archivos que se hayan modificado o creado en el directorio actual
  • git rm --cached <file> Quita el archivo del area de seguimiento, pero lo mantiene en el WORKING DIRECTORY. Este comando se utiliza cuando no existe una referencia para reestablecer el contenido.
  • git restore <file> Quita el archivo del INDEX, y lo reestablece a como estaba en la ultima cabecera.

Repository (Local / Remote)

Es el lugar donde se almacena las capturas en formato BLOB (Binary Object File) a las cuales podemos acceder mediante el historial de cambios. Para generar estos archivos debemos emitir un commit, el mismo valida los cambios y los almacena en la base de datos de GIT.

  • git commit Abre el editor para generar una confirmación de cambios que sera guardada en la base de datos del repositorio. Un uso rapido de este comando es git commit -m "message" que permite generar un commit con mensaje sin abrir el editor.
  • git diff <ref> Compara el WORKING DIRECTORY con el commit al cual apunta la referencia especificada. una forma de comparar cambios realizados cuando no se realizó un commit es git diff HEAD, pero si queremos saber como difiere nuestra cabecera con respecto a otro commit podemos indicarlo de diferentes maneras.
    • referencia absoluta: especificamente hablando es el número de HASH del commit que queremos comparar, por ejemplo c215714
    • referencia relativa indica cuantos commit atrás está nuestra referencia, por ejemplo HEAD^^ indica dos ancestros antes de HEAD y HEAD~4 indica 4 commits antes de HEAD
  • git blame <file> muestra las lineas de codigo del archivo especificado, colocando al principio en numero de commit, el nombre del autor, la fecha y hora en que se realizó el mismo, e incluso indica cuando los cambios todavia no fueron confirmados.
  • git remote lista los repositorios remotos que hayan sido agregados, si nuestro proyecto es un clon por defecto se agrega un remoto llamado origin, pero si fue creado, debemos hacerlo manualmente usando el comando git remote add <alias> <URL>

Apuntadores de GIT

HEAD

Es un apuntador móvil que nos permite navegar en el historial de cambios, normalmente apunta al último commit de la rama donde nos encontremos y podemos cambiar su posicion para poder visualizar el proyecto a como se encontraba en algun punto de referencia (commit) utilizando el comando git checkout <ref>

BRANCH

Las ramas son apuntadores dinámicos que siempre hacen referencia al último commit dentro del historial. Podemos crear diferentes ramas y cada una apuntará a un commit particular, este puede ser el mismo o pueden ser totalmente diferentes. Si se quiere comparar, podrian interpretarse como líneas de tiempo paralelas; las ramas son necesarias ya que permiten al apuntador HEAD navegar en el historial y volver al último commit sin problemas. Para crear una rama debemos utilizar el comando git branch <branch>

  • git branch <branch> crea una rama con el nombre espeficicado.
  • git branch -d <branch> elimina la rama seleccionada siempre y cuando los cambios de la misma hayan sido integradas a otra rama. (eliminación segura)
  • git branch -D <branch> elimina la rama selecciona aunque los cambios de la misma no se hayan integrado al proyecto
  • git merge <branch>

STASH

Es la zona temporal de cambios, donde podemos apilar todas las modificaciones realizadas en el WORKING DIRECTORY de aquellos archivos que se encuentren en el INDEX. Estos cambios no se almacenan en el repositorio remoto y utilizan como referencia el INDEX y el COMMIT al que apunta para poder reestablecer el directorio de trabajo, esto se hace cuando queremos cambiar de rama o commit dentro de la misma rama y tenemos cambios pendientes que no fueron cometidos. Para aplicarlo debemos utilizar los comandos git stash para limpiar el INDEX y almacenar los cambios en la ZONA STASH y luego git stash pop para aplicarlos y eliminar la referencia de la ZONA STASH.

  • git stash agrega los cambios que se encuentren en el index a la zona stash
  • git stash list muestra la lista de cambios almacenados en la zona stash
  • git stash apply aplica el último cambio almacenado en la zona stash
  • git stash drop elimina el último cambio de la zona stash
  • git stash pop aplica y luego elimina el último cambio que se encuentre en la zona stash

TAG

Las etiquetas son apuntadores estáticos que apuntan a un commit en particular dentro del historial, se suelen utilizar para el versionado de nuestro proyecto e incluso se pueden generar anotaciones al igual que en cualquier mensaje de commit. Se utilizan para facilitar la navegación dentro del historial a un punto en particular y normalmente se identifican de manera similar. Para subir los tags de un proyecto a un repositorio remoto debemos utilizar el comando git push <remote> --tags

  • git tag <tag> agrega una etiqueta al commit actual
  • git tag -d < tag> elimina la etiqueta seleccionada
  • git tag -l muestra todas las etiquetas creadas
  • git push <remote> --tags envia al repositorio remoto las etiquetas locales
